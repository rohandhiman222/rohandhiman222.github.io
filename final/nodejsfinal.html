# How can you handle uncaught exceptions in Node.js?

Answer:

Handling uncaught exceptions in Node.js is crucial for maintaining application stability. You can use the process object's uncaughtException event to catch and handle these exceptions.

Example:

process.on("uncaughtException", (err) => {
  console.error("Uncaught Exception:", err);
  // Perform cleanup and exit process
  process.exit(1);
});

throw new Error("This will be caught by uncaughtException");

Important Notes:

  • Using uncaughtException is considered a last-resort mechanism. It is better to use proper error handling practices within your code.

  • The process should typically be restarted after an uncaught exception since the application state might be corrupted.

# 6. Explain middleware in Express.js.

Answer:

Middleware in Express.js is a function that has access to the request object (req), the response object (res), and the next middleware function in the application's request-response cycle. Middleware functions can perform various tasks such as executing code, modifying the request and response objects, ending the request-response cycle, and calling the next middleware function.

Types of Middleware:

  • Application-level Middleware: Bound to an instance of the Express application using app.use() or app.METHOD().

  • Router-level Middleware: Bound to an instance of express.Router().

  • Error-handling Middleware: Defined with four arguments: (err, req, res, next).

  • Built-in Middleware: Provided by Express.js, such as express.static() for serving static files.

  • Third-party Middleware: Middleware created by the community, such as body-parser, morgan, etc.

const express = require("express");
const app = express();

// Application-level middleware
app.use((req, res, next) => {
  console.log("Time:", Date.now());
  next();
});

// Router-level middleware
const router = express.Router();
router.use((req, res, next) => {
  console.log("Router-level middleware");
  next();
});

// Route handling
app.get("/", (req, res) => {
  res.send("Hello World");
});

// Error-handling middleware
app.use((err, req, res, next) => {
  console.error(err.stack);
  res.status(500).send("Something broke!");
});

app.listen(3000, () => {
  console.log("Server is running on port 3000");
});

# 7. What is the difference between require and import in Node.js?

Answer:

Both require and import are used to include modules in a Node.js application, but they are used in different module systems.

require:

  • Used in CommonJS module system.

  • Synchronously loads modules.

  • Available in Node.js from the beginning.

  • Example: const fs = require('fs');

import:

  • Used in ES6 module system.

  • Asynchronously loads modules.

  • Requires "type": "module" in package.json or using .mjs file extension.

  • Example: import fs from 'fs';

  • Example:

// CommonJS (require)
const fs = require("fs");
fs.readFileSync("file.txt", "utf8");

// es6

// ES6 (import)
import fs from "fs";
fs.promises.readFile("file.txt", "utf8");

In Node.js, CommonJS is the default module system, but ES6 modules are increasingly supported and used.

# 8. How does the async/await syntax work in Node.js?

Answer:

The async/await syntax in Node.js is syntactic sugar built on top of Promises, making asynchronous code look and behave more like synchronous code, which is easier to read and write.

How it works:

  • async Function: Declares a function that returns a Promise. When the function is called, it returns a Promise.

  • await Keyword: Pauses the execution of the async function and waits for the Promise to resolve. It can only be used inside async functions.

const fetchData = async () => {
  try {
    const response = await fetch("https://api.example.com/data");
    const data = await response.json();
    console.log(data);
  } catch (error) {
    console.error("Error fetching data:", error);
  }
};

fetchData();

In this example, fetchData is an async function. It uses await to pause execution until the Promise returned by fetch is resolved, making it easier to handle asynchronous operations.

# 9. Explain the concept of middleware chaining in Express.js.

Answer:

Middleware chaining in Express.js refers to the sequential execution of middleware functions for handling HTTP requests. Middleware functions can perform tasks such as logging, authentication, parsing request bodies, and more. Each middleware function has access to the request (req), response (res), and the next middleware function in the chain.

How it works:

  • Middleware functions are added using app.use() or app.METHOD().

  • The next function is called to pass control to the next middleware in the chain.

  • If next is not called, the request-response cycle ends, and no further middleware functions are executed.

const express = require("express");
const app = express();

// Middleware function 1
app.use((req, res, next) => {
  console.log("First middleware");
  next();
});

// Middleware function 2
app.use((req, res, next) => {
  console.log("Second middleware");
  next();
});

// Route handler
app.get("/", (req, res) => {
  res.send("Hello World");
});

app.listen(3000, () => {
  console.log("Server is running on port 3000");
});

# 10. How can you improve the performance of a Node.js application?

Answer:

Improving the performance of a Node.js application involves various strategies, including optimizing code, managing resources efficiently, and leveraging Node.js features.

Strategies:

  1. Use Clustering: Utilize the cluster module to create multiple worker processes to handle concurrent requests.

  2. Asynchronous Operations: Use asynchronous methods to avoid blocking the event loop.

  3. Load Balancing: Distribute the load across multiple servers or instances.

  4. Caching: Implement caching mechanisms (e.g., Redis) to reduce the load on the database.

  5. Optimize Database Queries: Use indexes, optimize queries, and use connection pooling.

  6. Use Compression: Enable gzip compression for HTTP responses.

  7. Minimize Middleware: Use minimal and efficient middleware to reduce overhead.

  8. Profiling and Monitoring: Use tools like New Relic, PM2, or built-in Node.js profiler to identify and resolve performance bottlenecks.

  9. Memory Management: Monitor and manage memory usage to prevent memory leaks.

const cluster = require("cluster");
const http = require("http");
const numCPUs = require("os").cpus().length;

if (cluster.isMaster) {
  for (let i = 0; i < numCPUs; i++) {
    cluster.fork();
  }

  cluster.on("exit", (worker, code, signal) => {
    console.log(`Worker ${worker.process.pid} died`);
  });
} else {
  http
    .createServer((req, res) => {
      res.writeHead(200);
      res.end("Hello World\n");
    })
    .listen(8000);
}

In this example, clustering is used to fork multiple worker processes to handle incoming requests, improving the application's ability to handle concurrent connections.

# 1. What is the purpose of the buffer module in Node.js?

Answer:

The buffer module in Node.js is used to handle binary data. Buffers are used to read and write binary data, such as reading files, dealing with TCP streams, and interfacing with binary protocols.

Key Points:

  • Buffers provide a way to handle raw binary data directly in JavaScript.

  • Buffers are instances of the Buffer class in Node.js.

  • The buffer module is globally available, meaning it doesn't need to be explicitly required.

Example:

// Creating a Buffer from a string
const buf = Buffer.from("Hello World");
console.log(buf); // <Buffer 48 65 6c 6c 6f 20 57 6f 72 6c 64>

// Writing to a Buffer
buf.write("Node.js", 6);
console.log(buf.toString()); // Hello Node.js

// Reading from a Buffer
const buf2 = Buffer.alloc(10);
buf2.write("Node.js");
console.log(buf2.toString("utf8", 0, 6)); // Node.js

In this example, a buffer is created from a string, modified, and read using various buffer methods.

# 16. What is the difference between spawn, exec, and fork methods in the child_process module?

Answer:

The child_process module in Node.js provides methods to create child processes. The key methods are spawn, exec, and fork, each serving different purposes.

spawn:

  • Launches a new process with a given command.

  • Suitable for long-running processes.

  • Returns a ChildProcess object with stdout and stderr streams.

const { spawn } = require("child_process");
const ls = spawn("ls", ["-lh", "/usr"]);

ls.stdout.on("data", (data) => {
  console.log(`stdout: ${data}`);
});

ls.stderr.on("data", (data) => {
  console.error(`stderr: ${data}`);
});

ls.on("close", (code) => {
  console.log(`child process exited with code ${code}`);
});

exec:

  • Runs a command in a shell and buffers the output.

  • Suitable for short-lived processes.

  • Provides a callback with stdout and stderr data.

Example:

const { exec } = require("child_process");

exec("ls -lh /usr", (error, stdout, stderr) => {
  if (error) {
    console.error(`exec error: ${error}`);
    return;
  }
  console.log(`stdout: ${stdout}`);
  console.error(`stderr: ${stderr}`);
});

fork:

  • Creates a new Node.js process and establishes communication with it.

  • Suitable for spawning Node.js scripts as child processes.

  • Returns a ChildProcess object with IPC (Inter-Process Communication) channel.

Example:

const { fork } = require("child_process");
const child = fork("child.js");

child.on("message", (msg) => {
  console.log("Message from child:", msg);
});

child.send({ hello: "world" });

Differences:

spawn vs exec: spawn streams data, while exec buffers data. spawn is for long-running processes, exec is for short-lived processes.

fork: Specifically for Node.js scripts, with IPC for communication.
